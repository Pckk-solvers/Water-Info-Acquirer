# Hydrology Graphs Platform 要件

## 1. 目的

Parquet に保存済みの時系列データから、イベント系・経年系の水文グラフを再生成する。

本機能は、ユーザーが GUI 上で調整したスタイルを JSON として保存・再利用し、複数観測所・複数基準日のバッチ出力を行えることを目的とする。

## 2. 対象スコープ

### 対象

- Parquet ディレクトリの指定とスキャン
- 全6種グラフの生成
- 複数観測所 × 複数基準日のバッチ生成
- スタイル JSON の保存・読込・適用
- 基準線 CSV/JSON の読込と重畳
- ランチャー統合（起動可能にする要件定義）

### 非対象

- 外部サイトへの再取得
- Parquet 契約の再設計（`parquet-contract.md` を準拠元とする）
- Excel/PDF/SVG 出力（初期は PNG のみ）

## 3. 作成対象グラフ（全6種）

### 3.1 イベント系（時間変化）

1. ハイエトグラフ（降雨柱状図）
- 対象: 時間雨量・累加雨量
- 表示: 上から下へ垂れ下がる棒グラフ（Y軸反転）

2. ハイドログラフ（流量波形）
- 対象: 流量
- 表示: 折れ線 + 基準線（例: 計画高水流量）

3. 水位波形（水位ハイドログラフ）
- 対象: 水位
- 表示: 折れ線 + 基準線（例: 計画高水位(HWL), 避難判断水位）

### 3.2 経年系（年別変化）

4. 年最大雨量の経年変化図
- 対象: 各年の最大雨量
- 表示: 年別棒 + 水平基準線（例: 計画降雨量, 平均値）

5. 年最大流量の経年変化図
- 対象: 各年の最大流量
- 表示: 年別棒 + 水平基準線（例: 整備計画流量）

6. 年最高水位の経年変化図
- 対象: 各年の最高水位
- 表示: 年別棒 + 水平基準線（例: 計画高水位(HWL)）

## 4. 機能要件

### R-01 Parquet 入力契約

- 入力 Parquet は `docs/dev/hydrology-graphs-platform/parquet-contract.md` 準拠を必須とする。
- 読込時に必須列不足・許容値外を検出した対象はスキップし、理由を結果に表示する。

### R-02 イベント窓定義（固定）

- イベント系 3 種の窓は「日単位固定窓」とする。
- 基準日は窓の中央日として扱う（3日: 前日/当日/翌日, 5日: 前2日/前1日/当日/翌1日/翌2日）。
- 3日窓は 72 時間（3日 × 24 時間）、5日窓は 120 時間（5日 × 24 時間）。
- 欠損許容は 0 とし、窓内に 1 点でも欠損がある対象は生成不可（スキップ）とする。

### R-03 イベント系のメトリクス条件

- ハイエトグラフは `metric=rainfall`, `interval=1hour` を必須とする。
- ハイドログラフは `metric=discharge`, `interval=1hour` を必須とする。
- 水位波形は `metric=water_level`, `interval=1hour` を必須とする。

### R-04 経年系の算出条件

- 年最大値は「瞬間最大」を採用する。
- 経年系 3 種の生成最小年数は 10 年とする。
- 10 年未満の対象は生成不可（スキップ）とし、理由を表示する。

### R-05 バッチ処理方針

- 複数観測所 × 複数基準日の組み合わせで一括生成できること。
- 失敗時は全体中断せず、部分成功継続とする。
- 各対象の成功/失敗/スキップ理由を結果一覧で確認できること。

### R-06 基準線定義の入力

- 基準線は CSV/JSON 読込で与える。
- 紐付けキーは `source + station_key + graph_type` を必須とする。
- 基準線定義の最小項目は以下を必須とする。
  - `source`
  - `station_key`
  - `graph_type`
  - `line_name`
  - `value`
  - `unit`
- 詳細契約は `threshold-contract.md` に準拠する。

### R-07 スタイル JSON 管理

- スタイルは JSON として保存・読込できること。
- `schema_version` を必須とする。
- スタイル適用単位は「グラフ種別ごと」とする。
- GUI から保存・読込・初期化が行えること。
- GUI 上でプレビューを確認しながらスタイル調整できること。
- プレビュー表示中のスタイル変更は即時または短時間で再描画されること。
- 詳細契約は `style-contract.md` に準拠する。

### R-08 出力仕様

- 出力形式は PNG のみとする。
- 出力ディレクトリ構成は `観測所/グラフ種別/基準日` とする。
- 同一条件再実行時の上書きポリシーは「同名上書き」を初期要件とする。

### R-09 ランチャー統合

- 本機能は独立ツールとして、`src/water_info_acquirer/launcher.py` から起動可能であること。
- 起動導線は既存アプリと同方式（アプリ登録経由）で追加すること。

## 5. GUI 要件

### G-00 画面分割方針

- GUI は 1 画面詰め込みではなく 2 タブ構成とする。
  - 条件設定・実行タブ: 入力・選択・実行前検証・実行・停止・結果確認・ログ表示
  - スタイル調整タブ: スタイル編集・保存/読込・プレビュー確認
- タブ切り替え時は入力内容を保持すること。

### G-01 必須入力

- 条件設定・実行タブ
  - Parquet ディレクトリ
  - 基準線定義ファイル（CSV/JSON）
  - グラフ種別（6種）
  - 観測所（複数選択）
  - 基準日（複数選択: GUI手入力リストまたは CSV読込）
  - 3日/5日選択（イベント系対象）
- スタイル調整タブ
  - スタイル JSON（任意読込 + 編集 + 保存）
  - プレビュー対象（観測所・基準日・グラフ種別）

### G-02 実行前検証

- データ不足対象（欠損/年数不足/契約不一致）を実行前に一覧表示できること。
- 実行対象外の理由をユーザーが把握できること。
- 実行前検証は条件設定・実行タブで完結すること。

### G-03 実行と結果表示

- バッチ実行と停止操作は条件設定・実行タブに配置すること。
- 成功/失敗/スキップ件数を表示すること。
- 失敗・スキップは対象単位で理由を表示すること。
- 出力先パスを確認できること。
- 実行ログを条件設定・実行タブで確認できること。

### G-04 スタイル調整タブ

- スタイル調整タブはスタイル調整とプレビューに専念し、実行機能を持たないこと。
- 選択対象のプレビュー画像を GUI 上で確認できること。
- 実行時はスタイル調整タブで編集中の最新スタイルを適用できること（保存の有無は問わない）。

## 6. 入出力インタフェース要件

### I-01 スタイル JSON（要件）

- `schema_version` を必須とする。
- 6種それぞれに設定ブロックを持てる構造であること。
- 少なくとも以下を持てること。
  - 配色
  - 線種/線幅
  - フォント
  - 軸・凡例
  - 余白・サイズ
- 具体スキーマと値制約は `style-contract.md` を正本とする。

### I-02 基準線 CSV/JSON（要件）

- `source + station_key + graph_type` で基準線適用対象を決定する。
- 複数基準線を同一グラフに重畳できること。
- 仕様不一致レコードは読み飛ばし、警告として結果に残すこと。
- 具体スキーマと値制約は `threshold-contract.md` を正本とする。

## 7. 受け入れ基準

1. 6種すべてで PNG を生成できる。
2. 複数観測所 × 複数基準日でバッチ生成できる。
3. 日単位固定窓（中央基準）で欠損1点でもあるイベント窓は生成されず、理由が表示される。
4. 経年データが 10 年未満の対象は生成されず、理由が表示される。
5. 基準線が `source+station_key+graph_type` で正しく重畳される。
6. スタイル JSON の `schema_version` 未指定時は適用拒否される。
7. 出力が `観測所/グラフ種別/基準日` 構成で保存される。
8. 部分失敗時でも他対象は継続生成される。

## 8. 既知の前提

- 時刻は JST 固定で扱う（timezone 列は持たない）。
- 基準線値そのものの業務妥当性は本機能の責務外とする。
- 本書は要件定義であり、詳細アルゴリズムは `design.md` で確定する。
