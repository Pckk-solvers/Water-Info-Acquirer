# Hydrology Graphs Platform GUIレイアウト設計

## 1. 目的

- `requirements.md` と `design.md` を実装に落とすため、GUI の画面構成・操作・状態遷移を固定する。
- 対象は Tkinter 実装のレイアウト仕様であり、Pencil の作図手順は本書の範囲外とする。
- 1画面詰め込みを避け、2タブ構成で操作負荷を下げる。

## 2. 画面構成（2タブ）

### 2.1 全体方針

- GUI は同一 Toplevel 内で Notebook 2タブを切り替える。
- タブ構成:
  - `条件設定・実行`
  - `スタイル調整`
- 下部に共通ステータスバーを置き、状態・進捗・最終メッセージを表示する。

### 2.2 タブ1: 条件設定・実行

- 目的: 実行対象を確定し、実行前検証とバッチ実行、実行結果確認までを完結させる。
- レイアウト: 2カラム
  - 左: 入力フォーム + 実行前検証結果
  - 右: 実行結果（進捗/一覧/ログ）

左カラム（入力フォーム）:
1. `Parquet入力`
2. `基準線定義`
3. `グラフ種別`
4. `観測所選択`
5. `基準日設定`
6. `イベント窓設定（3日/5日）`

右カラム（検証結果）:
1. `実行進捗`
2. `対象別結果一覧`
3. `詳細ログ`

タブ1 操作:
- `実行前検証`
- `バッチ実行`
- `停止`

### 2.3 タブ2: スタイル調整

- 目的: プレビュー確認しながらスタイルを調整する。
- レイアウト: 2ペイン
  - 左: スタイル設定
  - 右: プレビュー

左ペイン（スタイル設定）:
1. `スタイルJSON 読込/保存/初期化`
2. `共通スタイル`
3. `グラフ種別スタイル`

中央ペイン（プレビュー）:
1. `プレビュー対象選択`（観測所、基準日、グラフ種別）
2. `プレビュー画像`
3. `プレビュー検証メッセージ`

タブ2 操作:
- `スタイル読込`
- `スタイル保存`
- `スタイル初期化`
- `プレビュー再生成`

## 3. 操作フロー

### 3.1 基本フロー

1. タブ1で Parquet をスキャンする。
2. グラフ種別・観測所・基準日・窓長を設定する。
3. 実行前検証結果を確認し、実行可能な条件を確定する。
4. タブ2でスタイルJSONを読込/調整し、プレビューで見た目を確認する。
5. タブ1でバッチ実行する（実行時に最新スタイルを適用）。
6. 実行結果を確認し、必要なら条件やスタイルを再調整する。

### 3.2 タブ遷移制約

- タブ切り替え時は入力値と選択状態を保持する。
- 実行中は条件入力の編集を禁止する。
- 実行中のタブ切り替えは許可するが、実行系操作はタブ1に限定する。

### 3.3 実行中制約

- 実行中は入力系ウィジェットを非活性化する。
- `停止` は未着手対象に反映し、完了済み対象は保持する。
- 停止後はタブ1で途中結果を保持し、再実行可能状態に戻す。

## 4. 状態遷移

- 画面状態は以下で固定する。
  - `idle`: 初期状態
  - `scanning`: Parquet スキャン中
  - `ready_config`: 条件設定・実行タブで編集可能
  - `ready_style`: スタイル調整タブで編集可能
  - `validating`: 実行前検証中
  - `running`: バッチ実行中
  - `completed`: 実行完了
  - `failed_partial`: 部分失敗終了

- 主な遷移:
  - `idle -> scanning -> ready_config`
  - `ready_config -> validating -> ready_config`
  - `ready_config -> running -> completed | failed_partial | ready_config(停止時)`
  - `ready_config <-> ready_style`

- 状態ごとの制御:
  - `ready_config`: 実行可能、条件編集可能
  - `ready_style`: スタイル編集・プレビュー可能
  - `running`: 実行以外無効、停止のみ有効

## 5. バリデーションと表示方針

### 5.1 実行前検証（タブ1）

- 以下を実行前に必ず検証し、NG対象は実行対象から外す。
  - Parquet 契約違反（必須列不足、許容値外）
  - イベント窓の欠損（欠損許容0）
  - 経年系の年数不足（10年未満）
- 検証結果はタブ1 左カラム内の検証領域に即時反映する。

### 5.2 エラー分類（タブ2）

- タブ2の読込/保存時に style JSON 契約検証（`schema_version` 含む）を実施する。
- 表示カテゴリは `design.md` と同一に固定する。
  - `contract_error`
  - `missing_timeseries`
  - `insufficient_years`
  - `threshold_not_found`
  - `style_error`
  - `render_error`
- UI表示は `カテゴリ + 人間可読メッセージ` の2段で示す。

## 6. スタイルJSON編集UI（タブ2）

- 編集単位は `共通設定` と `graph_type別設定` に分離する。
- 保存時は `schema_version` を必須検証し、不一致なら保存拒否する。
- 読込時は不正キーを警告表示し、適用可能キーのみ反映する。
- 初期化は「現在編集中のみ初期化」と「全グラフ種別初期化」を分ける。
- プレビューはスタイル変更から短時間で再描画し、描画失敗時も編集継続可能とする。
