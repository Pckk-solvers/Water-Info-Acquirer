# 観測所インデックス可用年メンテナンス 要件

## 1. 目的

JMA / WaterInfo の観測所インデックス JSON に、観測データの存在年を年リストとして保持し、定期更新可能なメンテナンスフローを整備する。

本機能の主目的は以下。

- 観測所インデックスの保守性向上
- 実行時オンライン判定の削減
- 定期更新可能な運用フローの確立

## 2. 対象スコープ

### 対象

- `scripts/` における観測所インデックス更新処理
- JMA / WaterInfo の可用年取得ロジックの整理
- インデックス JSON への可用年保存
- 定期更新を前提としたログ・途中保存・失敗継続方針

### 非対象

- 実データ Parquet からの開始日・終了日の厳密計算
- GUI での新規編集 UI
- 取得済みローカルデータの存在年とサイト上存在年の統合

## 3. 機能要件

### R-01 可用年の表現

- インデックス JSON では、データ存在期間を年粒度で保持すること
- 正本は範囲ではなく年リストであること
- 範囲情報が必要な場合は利用側で年リストから計算すること

### R-02 JMA 可用年保持

- JMA インデックス JSON には少なくとも `available_years_hourly` を保持できること
- 値は昇順・重複なしの整数年リストであること

### R-03 WaterInfo 可用年保持

- WaterInfo インデックス JSON には少なくとも `available_years_hourly` を保持できること
- WaterInfo インデックス JSON には `available_years_daily` も保持すること
- 値は昇順・重複なしの整数年リストであること
- 各観測所には `availability_updated_at` を保持すること

### R-04 更新モード

- 更新スクリプトは少なくとも以下のモードを持つこと
  - `rebuild`: 既存可用年を信用せず全件再取得
  - `update` または `resume`: 既存 JSON を引き継いで更新

### R-05 途中保存

- 全件更新時に途中結果を定期保存できること
- 中断時も途中成果が残ること
- 保存はテンポラリファイル経由の置換で破損しにくいこと

### R-06 部分更新

- 少なくとも開発・保守用途として、対象を限定した更新が可能であること
- 候補:
  - 都道府県絞り込み
  - 観測所コード指定
  - 件数制限

### R-07 失敗継続

- 一部観測所の取得失敗があっても、可能な範囲で更新を継続できること
- 失敗観測所はログまたはサマリで確認できること

### R-08 定期実行適性

- 定期実行時に過剰な負荷を避けるため、以下を調整可能とすること
  - timeout
  - request interval / sleep
  - max count

### R-09 実行入口

- 定期更新の代表入口は `scripts/refresh_station_indexes.py` とすること
- 個別スクリプトも単独実行可能であること

## 4. データ要件

### D-01 JMA データソース

- JMA の可用年判定は既存 `jma_availability.py` のロジックを再利用または移設すること
- 現時点では hourly を正式対象とすること
- JMA の可用年は PDF 由来情報を根拠にしないこと
- PDF は `start_date` や座標などの観測所属性補完専用とすること

### D-02 WaterInfo データソース

- WaterInfo の可用年判定は既存 `service.py` / `parser_availability.py` の availability 抽出処理を再利用または移設すること
- WaterInfo は `hourly` と `daily` の両方を正式対象とすること

### D-03 既存メタデータとの共存

- JMA の `start_date` など既存フィールドは保持されること
- `start_date` は可用年ではなく属性情報として扱うこと
- 可用年は既存メタデータを上書き破壊しないこと

## 5. 運用要件

### O-01 定期更新前提

- 本機能はエンドユーザ向け即時機能ではなく、保守者が定期的に回すメンテナンス処理として設計すること

### O-02 ログ

- 少なくとも以下をログに残すこと
  - 対象件数
  - 処理済み件数
  - 成功件数
  - 失敗件数
  - スキップ件数
  - 出力先

### O-03 スキーマ更新

- JSON スキーマに変更が入るため、`schema_version` もしくは同等の管理方針を明確化すること

### O-04 更新時刻の追跡

- 可用年の更新時刻は各観測所単位で追跡できること
- そのため `availability_updated_at` は観測所オブジェクトに保持すること

## 6. 設計判断

### J-01 正本は年リスト

- 範囲ではなく年リストを正本とする
- 理由: 欠年を保持できるため

### J-02 更新責務は scripts

- 外部サイトアクセス・更新ロジックは `scripts/` 側へ寄せる
- `src/river_meta` 側は原則として更新済み JSON を参照する
- WaterInfo のベース観測所生成と可用年更新は別スクリプトへ分離する

### J-03 利用側は JSON 優先

- 通常利用機能は JSON の可用年を参照する前提へ寄せる
- 最終方針として、可用年の参照元は更新済み JSON を正本とする
- オンライン判定は移行期間の暫定フォールバックに留める
