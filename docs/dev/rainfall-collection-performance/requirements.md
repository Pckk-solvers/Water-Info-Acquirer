# 雨量整理取得 高速化 要件（JMA / WaterInfo）

## 1. 目的
- 50年規模の取得で発生する長時間処理を短縮する。
- 既存機能（停止、Parquet互換、出力互換）を壊さずに段階的に高速化する。

## 2. 対象
- `src/river_meta/services/rainfall.py`
- `src/river_meta/rainfall/jma_adapter.py`（必要時）
- `src/river_meta/rainfall/waterinfo_adapter.py`（必要時）
- `src/jma_rainfall_pipeline/utils/http_client.py`（必要時）
- `src/water_info/infra/http_client.py`（必要時）
- GUI/CLI 設定項目（必要時）

## 3. 非対象
- 出力フォーマット（Excel/グラフ）の仕様変更
- Parquetスキーマ変更
- 取得元サイト仕様に依存する非公式手段の導入

## 4. 機能要件
### R-01 取得専用高速モード
- 条件: `export_excel=False` かつ `export_chart=False`
- 挙動:
  - 時系列集計 (`build_hourly_timeseries_dataframe`) を実行しない。
  - 年最大集計 (`build_annual_max_dataframe`) を実行しない。
  - 集計用DataFrameを生成しない。

### R-02 取得専用時の不要I/O削減
- JMA:
  - 月Parquet保存後の「年結合読み込み」を実行しない（取得専用時のみ）。
- WaterInfo:
  - 保存判定後、集計不要時はDataFrame変換を最小化する。

### R-03 並列取得（安全上限付き）
- 観測所単位で並列取得できること。
- 並列度は設定可能（例: 1〜8、既定は安全側）。
- 既定値は現行の取得先負荷を超えない安全値とする。

### R-04 HTTP制御の明示化
- 高速化時に、待機時間や同時実行数を設定から調整できること。
- JMA / WaterInfo で設定を分離可能であること。

### R-05 WaterInfo観測所名キャッシュ
- 同一観測所で年ループする場合、観測所名の再取得HTTPを抑止する。
- 1実行内でメモリキャッシュし、処理終了時に破棄する。

### R-06 既存挙動の維持
- 停止要求時の処理中断は維持する。
- 停止時も新規Parquetを保持する（削除しない）。
- 既存のParquet存在判定スキップは維持する。

### R-07 ログ可視化
- 実行ログに以下を出力:
  - 実行モード（通常/取得専用高速）
  - 並列度
  - 観測所ごとの処理時間
  - 全体処理時間

## 5. 非機能要件
### NFR-01 性能目標（相対）
- 同一環境・同一入力で、現行比で処理時間を短縮する。
- 最低目標:
  - 取得専用モードで現行比 30%以上短縮
  - 並列導入後に現行比 50%以上短縮（取得先制約内）

### NFR-02 安定性
- 429/5xxの再試行制御を維持し、失敗時に全体が異常終了しないこと。
- 観測所単位で失敗を隔離し、他観測所処理は継続すること。

### NFR-03 保守性
- 高速化ロジックはモード/設定で明示的に切り替え可能であること。
- テストで通常モードと高速モードの両方を検証できること。

## 6. 受け入れ条件
1. 取得専用高速モードで、集計関数が呼ばれないことをテストで確認できる。
2. 取得専用高速モードで、JMA年結合読み込みをスキップできる。
3. 並列度設定が有効で、並列度1で現行互換動作となる。
4. WaterInfo観測所名キャッシュが有効で、同一観測所の重複取得が削減される。
5. 停止・Parquet保持・Parquet存在スキップの仕様が維持される。
6. 主要シナリオの自動テストが追加され、回帰テストが通過する。

## 7. 実装順（推奨）
1. R-01, R-02（効果が大きく、リスクが低い）
2. R-05
3. R-03, R-04（段階導入、既定は安全側）
4. ログ整備・テスト拡充
