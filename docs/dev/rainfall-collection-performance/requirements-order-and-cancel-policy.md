# 雨量データ取得: 取得順制御 + 停止時Parquet保持 要件定義

## 1. 目的
- データ取得処理に「取得順序の切替」を導入し、運用ニーズに応じて処理順を選択可能にする。
- 停止要求時もParquetを保持し、途中取得結果を次回再開に活用できるようにする。
- GUI運用を前提にしつつ、将来運用のためCLIも同等仕様で提供する。

## 2. 背景（現状）
- 取得処理は `run_rainfall_analyze()` を利用している。
- JMA は月単位Parquetを逐次保存し、WaterInfo は年単位Parquetを保存する。
- 停止時ロールバック削除を行うと、途中取得結果を再利用できない。

## 3. 対象
- `src/river_meta/services/rainfall.py`
- `src/river_meta/rainfall/gui.py`（データ取得タブ）
- `src/river_meta/rainfall/cli.py`（必要に応じて）
- 関連テスト (`tests/river_meta/*`)

## 4. 非対象
- 出力（Excel/グラフ）仕様の変更
- Parquetスキーマ変更
- 取得元サイト仕様への依存追加

## 5. 機能要件
### R-01 取得順制御フラグ
- 取得順を制御するフラグを追加する。
- フラグ名: `collection_order`
- 値:
  - `station_year`: 観測所 -> 年の順に処理
  - `year_station`: 年 -> 観測所の順に処理
- 既定値は現行互換の `station_year`。
- 処理単位は `source × station_key × year` とする。
- 順序フラグは「処理順のみ」を変更し、以下は変更しないこと。
  - 対象観測所の解決結果
  - 年範囲フィルタ結果（JMA年可用性判定後の対象年集合）
  - 取得・保存ロジック（JMA月保存 / WaterInfo年保存）
- 同一入力・同一既存Parquet状態では、`collection_order` の違いで最終的に得られるParquet集合が一致すること。
  - ただし本条件は「停止なし・取得エラーなし」の完走実行を前提とする。
- `source=both` の場合でも順序規則は同一とし、`source` は安定順（`jma` -> `water_info`）で評価すること。
- 順序は再現可能であること（同じ入力で毎回同じ処理順になる）。
  - `station_year`: `(source, station_key, year)` の昇順
  - `year_station`: `(year, source, station_key)` の昇順
- 例:
  - 観測所A/B、対象年2023/2024 の場合
  - `station_year`: A-2023 -> A-2024 -> B-2023 -> B-2024
  - `year_station`: 2023-A -> 2023-B -> 2024-A -> 2024-B

### R-02 停止時Parquet保持（固定方針）
- 停止時に今回作成したParquetを削除しない。
- 停止時の部分取得結果は保持し、次回再実行時に再利用する。
- 停止時削除を切り替えるフラグは導入しない（方針固定）。

### R-03 欠け月再取得の継続
- JMA は月単位Parquet保存を維持する。
- 再実行時は既存のParquet存在確認を利用し、欠けている月のみ取得する。
- 停止時点までの保存済み月を次回再利用できること。

### R-04 年可用性判定との整合
- JMA年可用性判定の結果を適用したうえで処理対象年を決定する。
- `collection_order` を切り替えても、対象年集合の解釈が変わらないこと。

### R-05 UI/CLI反映
- GUIのデータ取得タブで以下を設定可能にする。
  - 取得順序（ラジオ）
- CLIでも取得順序フラグを受け取れること（必須）。
- CLIの `analyze` で `start_at/end_at` が指定された場合は、内部的に「開始年〜終了年（両端含む）」へ正規化して
  `collection_order` を適用すること。

### R-06 ログ要件
- 実行開始時に `collection_order` をログ出力する。
- 停止時に「新規Parquetを保持する」方針をログ出力する。

## 6. 非機能要件
- 既存のデフォルト挙動（順序）を維持する。
- 既存回帰テストを通過する。
- 設定値未指定時の挙動が一意である（既定値あり）。

## 7. 受け入れ条件
1. `collection_order=station_year` で現行同等の順序で処理される。
2. `collection_order=year_station` で年優先順に処理される。
3. 停止時に新規Parquetが保持される。
4. 停止後の再実行で、既存Parquet月をスキップし欠け月のみ取得される。
5. JMA年可用性判定は両順序で同じ対象年集合を導く。
6. GUI設定が実行引数へ正しく反映される。
7. `source=both` でも `jma -> water_info` の安定順が維持される。
8. CLI指定（`--collection-order`）が実行引数へ正しく反映される。
9. 既存の `tests/river_meta` 回帰が通過する。

## 8. 実装順（推奨）
1. `RainfallRunInput` に `collection_order` を追加
2. `run_rainfall_analyze()` のタスク生成を順序フラグ対応
3. 停止時ロールバック削除を廃止し、保持ログへ置換
4. GUI（必要に応じてCLI）に設定項目を追加
5. テスト追加・回帰実行

## 9. リスクと対策
- リスク: 順序切替で進捗ログが分かりづらくなる
- 対策: 観測所/年を常にログに明示する

- リスク: 停止保持運用で不完全年が増える
- 対策: 既存の欠け月検知ロジックで再実行時に自動補完する
