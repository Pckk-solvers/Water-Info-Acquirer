# 雨量・水位流量 時系列 Parquet 共通化 要件

## 1. 目的

JMA と水文水質DB（WaterInfo）の時系列データを、共通ルールで Parquet 保存できるようにする。

本要件の主目的は以下。

- 時刻の扱いを固定し、3日/5日グラフ再生成の基盤を安定化する
- 雨量だけでなく、水位・流量を同じ保存層で扱えるようにする
- 内部正本は Parquet、外部連携は JSON 出力で分離する

## 2. 対象スコープ

### 対象

- Parquet 保存スキーマの共通化
- 時刻正規化ルールの明文化と実装反映
- JMA / WaterInfo の保存前変換層
- グラフ再生成向けデータ抽出 API の前提定義

### 非対象

- 既存 GUI の全面改修
- 既存全 Parquet の一括再変換（段階移行とする）
- 新規 DB 導入

## 3. 機能要件

### R-01 時刻の正規保存ルール

- Parquet の正規時刻は `observed_at` に保存すること
- `observed_at` は timezone なし（naive）の JST として保存すること
- `observed_at` は JST 基準の 00:00〜23:00 を表すこと
- `1〜24時` 表記は保存せず、表示・出力時に変換すること
- source 差分吸収（例: JMA 1時間値の補正）は保存前正規化で一元処理すること

### R-02 タイムゾーンの明示

- 本要件では timezone 列は扱わない（非スコープ）
- 時刻解釈は `observed_at` を JST 固定として扱う
- 将来 UTC 連携が必要になった場合は別要件で拡張する

### R-03 共通スキーマ（縦持ち）

- 最低限、以下の列を持つこと
  - `source`
  - `station_key`
  - `station_name`
  - `observed_at`
  - `metric`
  - `value`
  - `unit`
  - `interval`
  - `quality`
- `metric` は少なくとも `rainfall`, `water_level`, `discharge` を扱えること
- `value` は各 `metric` の観測元1次値（非派生値）を保存すること
- 既存雨量列（`rainfall_mm`）との互換は移行期間中に吸収すること

#### スキーマ定義（必須列）

| 列名 | 型 | 必須 | 例 | 意味/保持ルール |
|---|---|---|---|---|
| `source` | string | 必須 | `jma`, `water_info` | データ取得元。利用側はこの値で分岐しない設計を原則とする。 |
| `station_key` | string | 必須 | `11_47401`, `106041286618100` | 観測所の内部識別子。保存・抽出の主キー要素。 |
| `station_name` | string | 必須 | `札幌`, `○○観測所` | 観測所名。未取得時は空文字可（`null` は使わない）。 |
| `observed_at` | datetime | 必須 | `2025-01-01 00:00:00` | 観測時刻（JST基準）。保存値は `00〜23時` 表現のみ。 |
| `metric` | string | 必須 | `rainfall`, `water_level`, `discharge` | 観測項目。列分岐ではなく metric で項目を識別する。 |
| `value` | float\|null | 必須 | `12.5`, `null` | 観測値。欠測は `null`。 |
| `unit` | string | 必須 | `mm`, `m`, `m3/s` | 観測値の単位。metric と整合していること。 |
| `interval` | string | 必須 | `10min`, `1hour`, `1day` | 観測間隔。既存互換のため当面文字列で保持。 |
| `quality` | string | 必須 | `normal`, `missing` | 品質区分。少なくとも `normal/missing` を扱う。 |

#### スキーマ定義（任意列）

- 任意列は本要件の実装対象外（非スコープ）とする
- 以下は将来拡張時の参考定義であり、今回実装での保存を要求しない
- 任意列を導入する場合は、別要件として追加合意してから実装する

| 列名 | 型 | 例 | 用途 |
|---|---|---|---|
| `raw_quality` | string\|null | `A`, `9`, `欠測` | 元データ固有の品質コード退避用。 |
| `prefecture_code` | string\|null | `11`, `1301` | 都道府県コード。source ごとに桁・体系が異なってよい（取得元仕様を尊重）。 |
| `prefecture_name` | string\|null | `埼玉県` | 都道府県名。取得時に得られる場合のみ保持。 |
| `raw` | object\|null | `{...}` | デバッグ・トレース用の元レコード断片。常時保存は必須にしない。 |

#### レコード例（雨量）

```json
{
  "source": "jma",
  "station_key": "11_47401",
  "station_name": "札幌",
  "observed_at": "2025-01-01T00:00:00",
  "metric": "rainfall",
  "value": 1.5,
  "unit": "mm",
  "interval": "1hour",
  "quality": "normal"
}
```

#### レコード例（水位）

```json
{
  "source": "water_info",
  "station_key": "106041286618100",
  "station_name": "○○観測所",
  "observed_at": "2025-01-01T00:00:00",
  "metric": "water_level",
  "value": 2.34,
  "unit": "m",
  "interval": "1hour",
  "quality": "normal"
}
```

### R-04 source 差分の吸収

- JMA / WaterInfo の差異は取得層ではなく保存前変換層で吸収すること
- 保存層以降（集計・グラフ）は source 非依存 API で扱えること
- 取得時に必要な識別子は source ごとに分離して扱うこと
  - JMA: 都道府県コード + 観測所コード（必要に応じて obs_type）
  - WaterInfo: station_id（都道府県は必須ではない）

### R-04.1 観測所メタ情報の扱い（参考・将来拡張）

- 本要件の実装スコープでは、観測所メタ情報（station master）の新規整備は必須としない
- 都道府県情報は時系列Parquetの必須列にはしない
- 将来、都道府県情報が必要な機能を追加する場合は、`source + station_key` で結合可能な観測所メタを別要件として定義する

### R-05 欠測・品質ルール

- 欠測は `value=null` と `quality` で表現すること
- 既存 quality（normal/missing）との互換マッピングを定義すること
- 元データ固有の品質情報がある場合は退避列で保持可能にすること

### R-06 保存単位

- 当面は現行互換のファイル粒度を維持すること
  - JMA: 観測所×年×月
  - WaterInfo: 観測所×年
- 粒度差は読込 API で吸収し、利用側に露出しないこと

### R-06.1 論理粒度と物理粒度の分離

- 利用側の論理粒度は source 非依存で統一すること
  - 例: `station_key + metric + interval + time_range` を指定して取得
- 物理保存粒度（ファイル分割）は source ごとの現行仕様を当面維持すること
- 利用側は「月ファイル/年ファイル」を意識せず、共通の読込 API だけを使用すること

### R-06.2 現行粒度を当面維持する理由

- 差分更新・再取得時の影響範囲を小さくできる
- 失敗時の復旧（再生成）コストを限定できる
- 既存処理との互換を保ちながら共通スキーマ移行を進められる
- 物理粒度の統一（例: 全sourceを観測所×年へ統一）は、運用実績で課題が顕在化した段階で別途判断する

### R-06.3 旧新スキーマ混在時の読込優先

- 旧スキーマと新スキーマが同一観測所・同一期間で混在する場合は新スキーマを優先すること
- 同一キーで衝突する旧スキーマ行は採用しないこと（新スキーマで上書き扱い）
- 旧スキーマは新スキーマが存在しない場合のフォールバックとしてのみ扱うこと

### R-07 JSON エクスポート

- 本要件では JSON エクスポートは非スコープとする
- Parquet を内部正本とし、JSON 仕様は別要件で定義する

### R-08 グラフ再生成前提

- 3日/5日グラフの入力は Parquet 抽出 API を唯一の入口にすること
- 表示用 `1〜24時` 変換は描画直前で行うこと
- 時刻軸は内部的に `00〜23` を維持すること

## 4. 非機能要件

### N-01 互換性

- 既存の雨量生成フローを即時破壊しないこと
- 旧スキーマ Parquet が混在しても読込時に判別・吸収できること

### N-02 保守性

- 時刻正規化ロジックを単一点に集約すること
- source 追加時に変換層の追加のみで対応可能な構造にすること

### N-03 性能

- 年単位データ読込が既存体感を悪化させないこと

## 5. 制約・注意点

### C-01 表示と保存の分離

- `24時` は表示慣習であり、保存表現ではない
- 保存に `1〜24` を持ち込まないこと

### C-02 WaterInfo 多変量

- 同一観測所で雨量・水位・流量が混在しうる
- 観測項目を列名で分岐せず、`metric` で識別すること

### C-03 既存Parquet移行

- 既存全量の一括変換は必須要件としない
- 新規取得から新スキーマ適用し、必要範囲を段階移行する

## 6. 未決事項

1. `interval` を文字列（`1hour` 等）で持つか、分単位整数（`interval_min`）へ統一するか
2. 旧 `rainfall_mm` スキーマ互換をいつまで維持するか

## 7. 推奨方針

第1段階は以下を採用する。

- 保存時刻は JST `00〜23` 固定
- 表示時のみ `1〜24` 変換
- 共通縦持ちスキーマ（`metric`,`value`,`unit`）へ拡張
- 保存前変換層で source 差分吸収
